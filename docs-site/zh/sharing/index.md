# 分享知识

Knowlery 把你明确审核过的一部分知识打包成可安装、可校验、可更新的**知识包
（bundle）**。它不是整个知识库的备份：只有你逐项批准的页面和来源会进入包中。

## 选择你的分享方式

| 我想要…… | 从这里开始 |
| --- | --- |
| 在 Obsidian 里点选、预览并发布 | [在 Obsidian 中分享](./obsidian) |
| 让 Codex、Claude Code 或其他 Agent 帮我操作 | [让 Agent 帮你分享](./with-agent) |
| 在终端完成审核与发布 | [发布知识包（CLI）](./publish) |
| 审核后由 CI 创建 GitHub Release | [使用 GitHub Actions 分发](./github-actions) |

无论使用哪个入口，底层都是同一套流程和同一份审核状态：

1. **导出与审核**——选定一个种子主题；范围内的每一页、每份原始来源都需要你逐条
   显式批准（未经审核的内容绝不出库）。
2. **[发布](./publish)**——`knowlery bundle publish` 在你配置的仓库里创建 GitHub
   Release，并交回一条可直接转发的完整消息（URL、校验和、受众）。默认私有；公开
   目标要过第二道门。
3. **[授权](./grant-access)**——发布**不等于**别人能访问。谁能安装取决于它放在
   哪里：公开托管（任何人）、私有仓库（collaborator 或组织成员）、或你直接发送的
   文件。
4. **安装**——[从 URL 安装](./install-from-url)或从本地文件安装。
5. **[订阅与更新](./subscribe-update)**——`check-updates` 向每个知识包的来源询问
   新版本；`update` 经同样的门安装它们。采用拉取模式，本地修改受保护。

::: tip 第一次分享
推荐先使用私有仓库，以 `0.1.0` 发布一个只包含少量页面的知识包。让另一台机器或一个
测试知识库完成一次“安装 → 查询 → 检查更新”，再扩大分享范围。
:::

## 团队推荐配置：组织货架

团队分享时，在你们的 GitHub 组织下建一个私有仓库——例如 `your-org/kb-bundles`——
并把组织的 base permission 设为 **Read**。

- 每位成员都能用自己的 GitHub 登录从它的 Release 链接安装。
- 权限跟随成员身份：入组即获得，离组即失效。无需逐人逐包管理。
- 团队里任何人都可以向同一货架发布自己的知识包——它自然成为团队知识中心。

GitHub 免费版即可（私有仓库和成员数均不限）。需要更细粒度时用 GitHub 的 team
配置——Knowlery 自身永远不管理权限。

## 安全姿态

- 知识包只包含**你逐条批准过的内容**——绝不会整库打包。
- Knowlery **从不存储或索要 token**。私有来源通过你自己的 `gh` CLI 登录或浏览器
  会话访问。
- 如果分享者在链接旁附了校验和，请校验下载：
  `knowlery bundle install <url> --verify <sha256>`。
