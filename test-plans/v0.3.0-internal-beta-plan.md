# Knowlery 0.3.0 Internal Beta Plan

## Purpose

Knowlery 0.3.0 is a product-level upgrade. It changes Knowlery from a one-time pre-configuration plugin into a personal knowledge review space for Obsidian. Before public release, the team should run an internal beta, collect structured feedback, and use that feedback to refine the product.

The internal beta should answer four questions:

- Do new users understand what Knowlery is for within the first minute?
- Do existing vault users know what to do next without studying source skills?
- Does the new `Today / This note / Weekly Review / Review Menu / System` structure feel natural inside Obsidian?
- Is the generated Knowledge Review Atlas useful enough for users to revisit?

## Branch And Build

- Branch: `release/v0.3.0`
- Internal beta version: `0.3.0-beta.1`
- Final public target version: `0.3.0`
- Obsidian minimum app version: `1.7.2`
- Release assets for testing: `main.js`, `manifest.json`, `styles.css`

Do not publish a GitHub release, tag, BRAT beta, or Obsidian Community Plugins submission until internal beta feedback has been reviewed.

## BRAT Distribution

Use BRAT for internal testing so beta testers can update consistently.

Recommended flow:

1. Build and verify `release/v0.3.0`.
2. Tag the tested commit as `0.3.0-beta.1`.
3. Let the GitHub release workflow create a prerelease with `main.js`, `manifest.json`, and `styles.css`.
4. Ask beta testers to install with BRAT from the Knowlery GitHub repository.
5. Collect feedback using the template below.
6. Publish follow-up beta tags such as `0.3.0-beta.2` only after fixes are merged into the release branch.

Tester install steps:

1. Install and enable `Obsidian42 - BRAT`.
2. Run `BRAT: Add a beta plugin for testing`.
3. Paste the Knowlery GitHub repository URL.
4. Choose the latest beta release.
5. Enable Knowlery from Community plugins.
6. Confirm the installed version is `0.3.0-beta.1`.

Use manual zip installation only as a fallback for testers who cannot use BRAT. If a zip is used, record exactly which beta version was installed.

## Beta Cohorts

### Fresh Vault Tester

Tester starts with a new or nearly empty vault.

Focus:

- First-run setup confidence
- Whether the user understands what Knowlery will create
- Empty-state guidance in Today
- Whether the first suggested action feels approachable

### Existing Vault Tester

Tester has an existing vault but has not used Knowlery before.

Focus:

- Whether Today gives a useful first maintenance direction
- Whether System diagnoses setup and vault health without feeling punitive
- Whether Review Menu explains workflows better than a skill list
- Whether the user can start a first review without reading documentation

### Returning User Tester

Tester has activity records from agent sessions or manual reflections.

Focus:

- Active trails shown in Today
- This note context and related activity
- Review Menu suggestions
- Weekly Review generation and report quality
- Claudian send/copy fallback behavior

## Required Test Scenarios

### 1. First Open

Steps:

1. Install the plugin in a test vault.
2. Open Knowlery from the ribbon and command palette.
3. Observe the initial screen.

Feedback:

- What do you think Knowlery helps you do?
- What would you click first?
- Is anything confusing or too technical?

Pass signal:

- Tester can explain the product as a knowledge review tool, not just a setup tool.

### 2. Today

Steps:

1. Test Today in an empty vault.
2. Test Today in an initialized vault with notes but no activity.
3. Test Today with recent `.knowlery/activity` records.

Feedback:

- Does Today clearly suggest one next move?
- Are stats useful or distracting?
- Does the tone feel warm rather than judgmental?

Pass signal:

- Tester understands the recommended next action without asking what a skill is.

### 3. This Note

Steps:

1. Open a Markdown note.
2. Switch to `This note`.
3. Copy the note prompt.
4. Send the note review to Claudian if available.

Feedback:

- Does Knowlery correctly identify the current note?
- Is the suggested review useful?
- Are `Copy note prompt` and `Send note review` clear?

Pass signal:

- Tester can use the current note as a natural review starting point.

### 4. Review Menu

Steps:

1. Open `Review Menu`.
2. Inspect suggested next moves.
3. Expand a static review move.
4. Expand Source skills only if needed.

Feedback:

- Do review moves feel like usable workflows?
- Is the source skills section appropriately hidden?
- Are copy/send labels clear?

Pass signal:

- Tester can choose a review move without needing to inspect raw skill prompts.

### 5. Weekly Review

Steps:

1. Open `Weekly Review`.
2. Generate the Knowledge Review Atlas.
3. Open `latest.html`.
4. Switch between English and Chinese.

Feedback:

- Is the report valuable, or does it feel like a mechanical activity dump?
- Does the Knowledge Hexagon reveal something useful?
- Do Timeline, Topic Constellation, and Knowledge Extensions help trace recent AI discussions?
- Is the default English version acceptable?
- Is the Chinese version good enough for daily use?

Pass signal:

- Tester would revisit the report after a week of real knowledge work.

### 6. System

Steps:

1. Open `System`.
2. Review diagnostics.
3. Expand configuration only if needed.

Feedback:

- Does System feel like maintenance, not the main product?
- Are warnings actionable?
- Is any health language too harsh?

Pass signal:

- Tester understands vault health without feeling blamed or overwhelmed.

## Triage Rules

- P0: blocks setup, dashboard open, data safety, build, or report generation.
- P1: blocks core product understanding or the Today -> action -> activity -> Weekly Review loop.
- P2: confusing wording, layout polish, minor missing state, report content quality issues.
- P3: nice-to-have enhancements for later 0.3.x releases.

## Release Gate

Do not proceed to public release until:

- All P0 issues are fixed.
- P1 issues are either fixed or explicitly deferred.
- `npm test` passes.
- `npm run build` passes.
- A real Obsidian smoke test passes on the Test vault.
- At least one fresh-vault tester and one existing-vault tester can explain Knowlery's value without being coached.
- The generated Knowledge Review Atlas is judged useful by at least one returning-user tester.

## Suggested Follow-Up After Beta

After feedback collection:

1. Group feedback by product area.
2. Decide what must be fixed for 0.3.0 vs deferred to 0.3.1.
3. Update release notes and official docs.
4. Run final release verification.
5. Only then prepare the public release PR, tag, and GitHub release.
