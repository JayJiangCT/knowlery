# Knowlery 0.6.0 — Deterministic Retrieval: Design & Usage

> Theme: **deterministic retrieval, measurable quality** — the core retrieval loop
> moved from "described in prompts" to "guaranteed by code and proven by evals".
> Accepted specs: `specs/0.6.0/`. Shipped as 0.6.1 (0.6.0 + community-review build
> compliance; functionally identical).

---

## Part I — Why this release exists

### The first-principles diagnosis

Knowlery's value reduces to one equation:

```
answer quality = f(evidence retrieved) x f(evidence freshness) / retrieval cost
```

Before 0.6.0, the retrieval side of that equation had a structural weakness: the
entire candidate-location pipeline lived in **prose inside the `/ask` skill** — a
six-step waterfall (read `INDEX.base`, four `obsidian properties` calls, read
`SCHEMA.md`, per-concept `obsidian search`, read the bundles registry, broad fallback).
Its correctness depended on an LLM obeying a 380-line prompt, it cost ~8 CLI round
trips per question, and it drifted across model versions. Worse, **nothing measured
it**: vault health checked structure (frontmatter present, links resolve), never
semantics (did `/ask` find the right pages?).

Three more problems followed from the same root:

1. **Freshness was solved at the wrong layer.** Detecting "a source note changed after
   the page citing it was written" is mechanical (mtimes + `sources` frontmatter), yet
   the only freshness story was an LLM-driven review workflow.
2. **Fixed context grew forever.** Every agent session imported `KNOWLEDGE.md` +
   `SCHEMA.md` + `INDEX.base`. The SCHEMA taxonomy tables are designed to grow with
   every `/cook` — a token tax with a monotonically rising rate.
3. **`/cook`'s incremental scope** hung off a single prompt-maintained timestamp in
   `log.md` — one failed cook loses the marker.

0.6.0 is the systematic answer: move deterministic work into code, keep judgment in
the model, and build the measuring instrument first.

### What was deliberately kept

- The **ownership boundary**: user notes are read-only to agents; agents own only
  `entities/`, `concepts/`, `comparisons/`, `queries/`.
- The compiled layer as **plain markdown + frontmatter** (auditable, no lock-in).
- **No embeddings.** At personal-vault scale, lexical + structured + graph signals
  suffice, and staying dependency-free is the low-barrier moat. The real gaps (aliases,
  cross-language) were closed by smarter deterministic mechanisms instead.

## Part II — Technical design

### F1. The evaluation harness (`evals/`)

Built first, because every later claim depends on it.

- **Golden set** (`evals/golden/questions.yaml`): 30 questions across 8 categories —
  entity-lookup, concept-lookup, synthesis, alias, bilingual, bundle, user-note,
  unanswerable. Scoring uses `must` pages only; `nice` is audit annotation.
- **Fixture vault** (`evals/fixtures/vault/`): a coherent fictional workspace
  (a team building the "Pulseboard" monitoring product) with deliberate traps —
  unrecorded aliases, Chinese sources compiled into English pages, bundle-only
  knowledge, never-compiled drafts, duplicate basenames, an orphan page.
- **Baseline retriever**: a deterministic, LLM-free emulation of the old `/ask`
  waterfall, frozen at `evals/reports/baseline.json`.
- **Metrics**: recall@5, recall@10, MRR per category; unanswerable accuracy separately.
- **CI guard**: `npm run eval -- --assert-baseline` fails any PR that regresses the
  frozen baseline or (since F2) the engine thresholds.

### F2. The retrieval engine (`src/core/query/`)

Pure TypeScript, zero `obsidian` imports (enforced by a purity unit test). Four
mechanisms, each mapped to a measured baseline weakness:

| Mechanism | What it does | Baseline gap it closes |
|---|---|---|
| Field-weighted scoring | title/aliases x4, tags/basename x3, description x2, body x1; light morphological variants (label↔labels, pick↔picked); CJK chunks matched as substrings with a 2x specificity multiplier | alias 0.500 |
| Source-graph boost | a matching raw note transfers 50% of its score to the compiled page citing it in `sources` | bilingual 0.250 (a Chinese question matches the Chinese source, credit flows to the English page) |
| Score-first ranking | tier is a multiplicative nudge (agent x1.15, bundle x1.10), not a gate | bundle recall@5 = 0 (hard tier ordering buried bundle knowledge) |
| Abstention | explicit `No confident matches` verdict when nothing hits structured fields and body evidence is weak | fabricated answers for unanswerable questions |

Plus a conservative **abbreviation prefix match** (terms ≥5 chars whose common prefix
covers ≥ max(4, len−2) chars of a title/basename token, at half credit) — how "colld"
reaches "collector-daemon".

There is deliberately **no cached index**: the engine live-scans (`scanVault`), because
a personal vault scans in milliseconds and a cache would reintroduce the staleness
class this release exists to eliminate.

**Result on the harness:** recall@10 0.778 → **0.926**, MRR 0.428 → **0.846**, alias
0.500 → 0.750, bilingual 0.250 → 0.750, bundle recall@5 0 → 1.0, no category
regression. The three remaining misses are honest: an unrecorded colloquial synonym, an
English question whose only answer is a fully-Chinese note, and one deliberate
unanswerable near-collision.

### F2/F5. Two transports, one engine

| Transport | When | How it works |
|---|---|---|
| `obsidian knowlery:query` / `knowlery:stale` | Obsidian running (1.12.2+, CLI enabled) | Registered via `Plugin.registerCliHandler`. The CLI host only captures handler output within the current microtask queue, so the handler cannot do I/O — it runs synchronously over a **live in-memory snapshot** (`LiveQuerySnapshot`) built in the background at layout-ready and updated incrementally from vault events. Memory-only; can't go stale while Obsidian is closed because it doesn't exist then. |
| `node .knowlery/bin/query.mjs` | Obsidian closed; needs only Node | A **self-contained script** (engine + gray-matter bundled, ~500KB) that live-scans. Built by an esbuild pre-pass into a committed generated module, embedded in `main.js` (community installs only distribute `main.js`), and written into the vault on setup and on every plugin upgrade. |

Page construction and output rendering are shared functions, so transport parity is
structural — asserted byte-identical by tests.

### F3. Mechanical staleness (`src/core/query/staleness.ts`)

Three deterministic definitions, no model, no persisted state:

- **Stale page** — an agent page where a cited source's mtime is *strictly greater*
  than the page's mtime (equal mtimes, e.g. one cook pass writing both, are fresh).
- **Uncooked note** — a user note (outside `Library/`) cited by no page's `sources`:
  candidate new material, newest first.
- **Dangling source** — a citation resolving to nothing: a data-quality signal,
  surfaced rather than guessed at.

Surfaces: `obsidian knowlery:stale` / `query.mjs --stale`, the dashboard's
**Knowledge health** section (counts, top stale pages, drill-in, "Copy re-cook
prompt"), and `/cook`'s incremental mode (replacing the `log.md` timestamp; the log
remains append-only history). System files (`KNOWLEDGE.md`, `SCHEMA.md`, `log.md`) are
excluded from all scans. LLM-driven freshness review remains the *second* tier, for
semantic supersession only.

Known trade-off (accepted): sync tools that rewrite mtimes can flag many pages at
once — the flag is a re-read hint for `/cook`, never an automatic action.

### F4. Fixed-context slim-down

Fixed context is now `KNOWLEDGE.md` + rules only.

| Content | Before | After |
|---|---|---|
| `KNOWLEDGE.md` (operating card) | injected | injected (unchanged) |
| `SCHEMA.md` (grows forever) | injected | on-demand read; explicit "read before creating/re-tagging pages" instruction |
| `INDEX.base` (Base view YAML) | injected | on-demand `base:query` tool; retrieval goes through the engine |

Existing vaults migrate non-destructively, once per version: exact-line removal of the
two stale imports in `.claude/CLAUDE.md`; an `instructions`-array filter in
`opencode.json` with a parse-failure bail-out. Re-adding the imports after upgrading
sticks.

## Part III — Usage guide

### For you (the vault owner)

- **Upgrade**: install 0.6.1+. Requires Obsidian 1.12.2+. The plugin auto-syncs
  updated skills and writes `.knowlery/bin/query.mjs` on first load after upgrade;
  `CLAUDE.md`/`opencode.json` are slimmed automatically (your own edits untouched).
- **Knowledge health** (dashboard): shows "N pages have changed sources · M notes never
  compiled". Drill in for the full list; **Copy re-cook prompt** produces a `/cook`
  prompt scoped to exactly the stale pages and their changed sources.
- **Optional**: enable Obsidian's command line interface (Settings → General) to get
  the in-app `knowlery:query` / `knowlery:stale` commands.

### For agents (what the skills now do)

- **`/ask`** runs **one** retrieval call instead of a six-step waterfall, reads the
  ranked candidates, synthesizes with citations. On `No confident matches` it tells you
  the vault lacks coverage and suggests `/cook` — it will not invent an answer.
- **`/cook`** with no arguments starts from the staleness report: re-read only the
  changed sources of stale pages; consider recent uncooked notes as new material.

### Command reference

```bash
# Retrieval (identical output, pick by environment)
obsidian knowlery:query question="why did median replace average?" [k=<n>] [json]
node .knowlery/bin/query.mjs "why did median replace average?" [--k <n>] [--json]

# Staleness
obsidian knowlery:stale [json]
node .knowlery/bin/query.mjs --stale [--json]

# Run the evaluation (repo checkout)
npm run eval                      # both retrievers, per-category scores
npm run eval -- --assert-baseline # the CI guard
```

Output anatomy (text mode):

```
1. concepts/response-time-metrics.md [concept] (score 38.4) — Latency is reported as median...
   evidence via source: Daily/2026-04-05.md, Projects/pulseboard-prd.md
```

`evidence via source:` means a raw note the page cites matched your question — read
those notes too when original context matters.

### Troubleshooting

| Symptom | Meaning / action |
|---|---|
| `Snapshot warming up — retry in a moment...` | The in-app snapshot is still building right after plugin load. Retry, or use the embedded script. |
| Large stale list after syncing devices | Sync rewrote mtimes. Expected; cook selectively. |
| Abstention rarely triggers on a big vault | By design: it fires on zero structured-field matches, so common words usually surface weak candidates instead. A score-quality threshold is a recorded 0.8 candidate. |
| A page you expect isn't found by the words you use | The name isn't written down. Add it to the page's `aliases` frontmatter (0.7.0's `/cook` does this automatically while compiling). |

### Files this release adds to a vault

| Path | What |
|---|---|
| `.knowlery/bin/query.mjs` | the self-contained retrieval/staleness script (managed; refreshed on upgrades) |
