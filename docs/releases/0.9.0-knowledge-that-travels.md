# Knowlery 0.9.0 — Knowledge That Travels: Design & Usage

> Theme: **knowledge that travels** — the sharing loop 0.8.0 completed mechanically
> gained its last mile: bundles move over the network, subscribers pull updates, and
> public publishing is guarded in proportion to its irreversibility. Accepted specs:
> `specs/0.9.0/`.
>
> **Status note:** F1 (remote install) and F2 (publish) are merged and
> maintainer-accepted; F3 (update & subscription) is implemented and in acceptance;
> F4 (retrieval specificity signal) is planned. This document tracks the accepted
> design and will be finalized at release prep.

---

## Part I — Positioning: the distribution model

0.8.0's sharing loop still ended with "get the zip onto the receiver's disk
yourself". 0.9.0 replaces that with URLs — under four binding principles decided
before any feature spec:

1. **Hosting is borrowed, never operated.** Bundles live on GitHub Releases (or any
   static host). Knowlery runs no registry, no server — the same leverage call as
   OIDC publishing, applied to distribution.
2. **The trust anchor is source continuity.** An installed bundle records its
   source; updates come only from it. Trusting a bundle = trusting its repo —
   structurally the same as a Homebrew tap.
3. **Knowlery never manages credentials.** Public sources fetch anonymously;
   private repos delegate to the receiver's own `gh` login (or their browser
   session). No tokens stored, configured, or seen. **`gh` is an accelerator, never
   a prerequisite** — every flow has a documented no-`gh` degradation.
4. **Pull, not push.** No daemon, no notifications; agents provide the cadence.

**The recommended team topology — the org shelf**: one private repo under a GitHub
organization (`your-org/kb-bundles`) with base permission Read. Membership *is* the
subscription permission (join → access, leave → revoked); anyone on the team
publishes to the same shelf; GitHub Free suffices. Zero Knowlery code — it's a
usage pattern of the credential-delegation design.

**Recorded for later (Beyond 0.9)**: a hosted platform (owners pay to publish,
subscribers subscribe, free-or-paid at the owner's choice) slots in as a second
*publish target* and *upstream* — the seams are kept clean now precisely so that
needs no redesign. Platform auth would live in a platform client layer, never the
core.

## Part II — Technical design

### F1. Remote install

- `bundle install <url>`: download to a temp file, then the **identical** local
  pipeline — same preview, conformance/version/path-safety gates by construction.
- **Three-tier source access**: anonymous fetch (public — `gh`-free end to end,
  redirects followed since GitHub assets 302 to storage) → automatic `gh` fallback
  for GitHub-shaped URLs refused anonymously (GitHub answers 404 for unauthorized
  private assets, never revealing existence) → browser-download guidance without
  `gh`.
- **`--verify sha256-…`**: transport integrity checked on the raw bytes *before*
  anything parses them; works for local files too.
- **The F3 seam**: the registry's `source` field records the original URL, not the
  temp path. Transport is shell-supplied (node http(s) for the CLI; Obsidian's
  `requestUrl` for the plugin — a lint rule against global `fetch` forced this and
  improved the design). Plugin parity: a URL field in the install modal.
- Safety clauses tested on a loopback HTTP server + a stub `gh` executable on PATH:
  same-gates-remote-or-local, verify-before-parse, no partial state on mid-download
  failure, plaintext-http warning, strict `gh` scoping, source-seam recording.

### F2. Publish flow

- `bundle publish <seed>`: the same review gate as export (unreviewed → checklist,
  exit 1 — a runner spy proves `gh` is never invoked), compile + zip, GitHub
  Release via the owner's `gh`, then the **audience statement**: who can install
  this and how to grant access — because *publishing does not grant access*, and
  that confusion ("the link 404s for my teammate") is the #1 troubleshooting entry.
- **Publisher-side configuration**: target repo remembered per bundle in
  `export-scope.json` (`publish: { repo, visibility }`); missing repos bootstrap
  `gh repo create --private` after confirmation. `--public` never flows into repo
  creation. The vault itself is never git-touched.
- **Default-private posture + the second gate**: `--public` re-lists every approved
  item carrying risk hints, states irreversibility ("caches, mirrors, and crawlers
  retain it even if deleted"), and requires re-acknowledgment — interactive
  `publish` phrase or explicit `--acknowledge-risks` (which agents may pass only
  after showing the user the list). Per-publish, never remembered.
- **Credential-pattern risk scanning**: `ghp_`/`github_pat_`/`sk-`/`AKIA`/`xox*`/
  private-key blocks, private IPv4 ranges, CN mobile shapes — evidence **redacted**
  (the hint proves the finding without repeating the secret). Shared scanner → the
  patterns surface at *export review* too.
- **No-`gh` degradation**: a guided manual web release with every value precomputed
  (zip path, `releases/new` URL, tag, sha256). The release it produces is
  indistinguishable from a `gh`-published one.
- Release notes embed a copy-ready `install --verify` line; **`--force` refreshes
  the notes along with the asset** (acceptance blocker: a replaced asset with a
  stale body checksum would poison trust in `--verify`).
- Tag convention: `<bundle-id>-v<version>` — multi-bundle shelves need the prefix.

### F3. Update & subscription

- **The upstream protocol** (plan-binding): abstract `latest()` →
  `{version, url} | unreachable | needs-auth`. GitHub Releases is the only v1
  implementation (F1's transport tiering; tags filtered by the bundle's own
  `<id>-v` prefix so shelves never cross-contaminate). One shared `compareVersions`
  (stable dotted-numeric contract; `1.10 > 1.9`).
- **`check-updates`**: strictly read-only (proven by a write-spy), exit 0 across
  all five outcomes — available / current / `unchecked` (no version protocol —
  bare zip URLs) / `skipped` (private, `gh` needed) / unreachable. Absence of
  checkability is a result, not an error.
- **`update <id> | --all`**: rides the F1 install pipeline on the new asset URL.
  **Local-modification protection**: install now records per-file hashes, so the
  check names exactly which files were `(edited)`/`(added)`/`(deleted)`; refuses
  without `--force`. Reinforces reference-don't-edit.
- **Staged replacement** (spec-review correction — the old install did
  rmdir-then-write, losing the installed copy on mid-write failure): gates pass →
  write to `Library/.staging-<id>/` → swap via `.old-<id>` backup → drop backup.
  Pre-swap failure leaves live untouched; mid-swap failure keeps a named backup.
  Needed a new `VaultFs.rename` primitive. Benefits *all* installs, not just
  updates.
- Dashboard: Check-updates + per-bundle Update buttons; no check on load.

### F4. Retrieval specificity signal (planned)

Closes 0.8's exactly-half coverage boundary: distinguishing "legitimate half
coverage" from "collision half coverage" needs a signal on the *unmatched* terms
(a high-specificity CJK chunk missing means something different from English
filler missing). Seed cases from the 0.8 acceptance round (斑马/莫言/胚胎筛查/鼠疫)
enter the golden set; same eval discipline as 0.8 F2.

### Documentation as a gated deliverable

The docs site gained a top-level **Sharing Knowledge** section (en + zh), shipped
chapter-by-chapter *with* each feature: journey overview (publish ≠ grant access),
Publish, Grant Access (the audience table + org-shelf recipe), Install from a URL,
Subscribe & Update, and a troubleshooting page whose first entry is the 404
confusion. The `knowlery-cli` skill is the machine-readable counterpart — each
feature added conduct rules (verify: never fabricate a checksum; publish: never
`--public` on the agent's initiative; update: never without the user asking).

## Part III — Usage guide

### Owner: publish once, then it's one command

```bash
knowlery bundle publish drone-delivery --repo your-org/kb-bundles   # first time
# → Published jay.drone-delivery v1.2.0 to your-org/kb-bundles (private).
#   Who can install: members with read access to your-org/kb-bundles.
#   Share:  knowlery bundle install https://github.com/.../pack.zip --verify sha256-…

knowlery bundle publish drone-delivery                              # every time after
```

New version: re-export with `--bundle-version 1.3.0`, publish again.

### Subscriber: install once, then pull

```bash
knowlery bundle install https://github.com/your-org/kb-bundles/releases/download/... --verify sha256-…
knowlery bundle check-updates
# jay.drone-delivery  v1.2.0 → v1.3.0 available
knowlery bundle update jay.drone-delivery
```

Private shelf? Your own `gh auth login` (or browser download) is the credential —
org members have access by membership.

### FAQ

| Question | Answer |
|---|---|
| Does publishing make my bundle reachable? | No — publishing and granting access are separate steps. The publish output tells you exactly who can install and how to grant more. |
| Can I publish/install without `gh`? | Public installs never need it. Private installs: browser download + local install. Publishing: a guided manual web release (~1 minute). |
| What if I edited an installed bundle? | `update` refuses, naming your edited files. Move your notes into your own pages (reference, don't edit), or `--force` to overwrite. |
| Can updates lose my installed version? | No — replacement is staged; any failure leaves the old version in place (worst case: a named backup dir to rename back). |
| Is there a knowledge marketplace? | Not yet — recorded as the Beyond-0.9 trajectory; today's design keeps the seams (publish target, upstream protocol) it would slot into. |
