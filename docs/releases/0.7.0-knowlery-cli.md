# Knowlery 0.7.0 — The `knowlery` CLI: Design & Usage

> Theme: **one core, two shells** — the knowledge-base lifecycle became a standalone
> CLI (`npm i -g knowlery`) over exactly the same workspace format the Obsidian plugin
> uses. Accepted specs: `specs/0.7.0/`.

---

## Part I — Positioning: what the CLI is (and is not)

The CLI is **not** "Knowlery without Obsidian". It is a second shell over one shared
core:

| Shell | Gets you |
|---|---|
| **Obsidian plugin** | Everything: review space, Knowledge health UI, live in-app retrieval (`knowlery:query`), Obsidian's semantic layer (wikilink resolution, backlinks, Bases) — plus all lifecycle operations |
| **`knowlery` CLI** | `init` / `sync` / `health` / `query` / `stale` / `bundle install\|list\|uninstall` — for terminals, cloud agents, CI, and machines without Obsidian |

The contract between them is the **workspace format**, not the tools: a folder
initialized by the CLI opens in Obsidian with zero migration, any Knowlery vault works
with the CLI as-is, and both shells may operate on the same vault concurrently (all
writes are idempotent and content-compared).

Why this was cheap to build: 0.6.0 already forced the hardest subsystem (retrieval)
to be Obsidian-free. 0.7.0 extended the same discipline to the rest of the lifecycle.
BYOAO — the original "global CLI install" experiment — is archived; this supersedes it.

## Part II — Technical design

### F1. The `VaultFs` inversion

The lifecycle logic (init, skill/rule sync, migrations, health, bundle install) used
to do I/O through Obsidian's `App`. F1 inverted it onto a minimal interface:

```ts
interface VaultFs {
  exists / read / readBinary / write / writeBinary
  mkdir / remove / rmdir / list          // vault-relative paths, '/' separators
}
```

Two implementations:

- **`obsidianVaultFs(app)`** — wraps the vault API, preserving the pre-existing write
  semantics verbatim (vault API first for indexed files so the metadata cache stays
  coherent; adapter fallback for hidden paths).
- **`nodeVaultFs(root)`** — plain `node:fs/promises`, used by the CLI and by tests.

Guarantees: a **purity unit test** forbids `obsidian` value imports across the whole
inverted set (17+ modules); a flagship test runs the full `executeSetup` against
`nodeVaultFs` in a real temp directory and asserts the complete file tree; the entire
refactor shipped with **zero behavior change** (all prior test assertions unmodified,
eval baseline untouched).

### F2. The CLI shell and packaging

- `src/cli/` is deliberately thin: argv parsing, readline prompts (TTY-detected;
  non-TTY with missing flags errors deterministically, naming the exact flag), output
  rendering. Command handlers take a `VaultFs` and are unit-tested without spawning.
- **One sync list, two shells**: `runVaultSync(fs, platform, toolVersion)` is called
  by both the plugin's upgrade hook and `knowlery sync` — the shells structurally
  cannot drift. A `loggingVaultFs` wrapper lets `sync` report exactly which files
  changed; every write is content-compared, so a second `sync` is a true no-op.
- **Init parity**: `knowlery init` runs the same `executeSetup` as the plugin wizard;
  a test initializes one workspace per shell and asserts the trees are identical
  file-by-file. (Optional tool installs — Claudian etc. — are wizard-only.)
- **Build & distribution**: esbuild produces three artifacts from one `src/` —
  `main.js` (plugin), the embedded `query.mjs` (generated module inside `main.js`),
  and `knowlery-cli.mjs` (npm-only; shebang + createRequire banner; the same
  scanner-hygiene shims). `package.json` has `bin: { knowlery }` and a `files`
  whitelist, so **npm ships exactly the CLI artifact + README + LICENSE** — never the
  plugin. The release workflow publishes to npm on version tags, gated on the
  `NPM_TOKEN` secret (skips with a warning when absent; prerelease tags go to the
  `beta` dist-tag).

### F3/F4. Query, staleness, and bundles in the CLI

- `knowlery query` / `knowlery stale` wrap the 0.6.0 engine through the shared
  renderers — output **byte-identical** to the in-app commands and the embedded
  script (spawn tests compare all transports on the same workspace). They are
  read-only live scans with **no init gate** (any markdown folder works).
- `knowlery bundle install <zip-or-folder>` runs the same `installBundle` core the
  plugin modal uses, with every 0.5.0 safety property intact: path-safety assertions,
  version gate (`--force`), conformance gate (`--skip-conformance` — the CLI form of
  the modal's explicit acknowledgement), registry update, `KNOWLEDGE.md` pointer
  block. `list`/`uninstall` complete the receiving side. Bundle **export** stays in
  the plugin (the approve/flag review gate is UI-shaped; a headless form is 0.8
  material). Bundle commands do require an initialized workspace.

### F5. Environment-adaptive skills

One skill text serves both shells:

- **Three-transport retrieval ladder** in `/ask` and `/cook`:
  `obsidian knowlery:query` → `knowlery query` (global CLI) →
  `node .knowlery/bin/query.mjs` (always present) → degraded prose last.
- **Headless write branch** in `/cook`, `/organize`, `vault-conventions`: Obsidian CLI
  stays the stated preference (it maintains wikilink-graph integrity); headless
  environments write files directly under identical conventions and run
  `knowlery health` after bulk changes.
- **Retrieval-aware compiling**: `/cook` records nicknames, abbreviations, and
  **cross-language titles** into `aliases` frontmatter — "a name that is not written
  down cannot be found" (the write-side closure of the 0.6.0 alias/bilingual gaps).
- **`/audit` on deterministic tools**: `obsidian orphans` / `unresolved` / `deadends`
  plus the staleness report; `danglingSources` is scan category 7.

### Versioning policy and the downgrade guard

- **Lockstep**: one version, one tag, one changelog; plugin (GitHub release assets)
  and CLI (npm) always release together. Semver: minor = features, patch = fixes,
  **major reserved for workspace-format breaking changes**. The separately-versioned
  contract is the workspace format (`.knowlery/manifest.json` `version`).
- **Downgrade guard**: the manifest records `lastSyncedBy` (the core version that last
  ran sync). A tool older than that refuses to sync — the plugin shows a Notice, the
  CLI exits 1 with an upgrade hint. No override flag by design: an older shell
  rewriting newer skill content is precisely the accident this prevents. Dev builds
  (unparseable versions) bypass the guard; legacy vaults without the field are always
  allowed.

## Part III — Usage guide

### Install

```bash
npm i -g knowlery        # requires Node 18+
knowlery --version
```

### Command reference

```
knowlery init   [--dir <path>] [--platform claude-code|opencode] [--name <kb name>] [--force]
knowlery sync   [--dir <path>]
knowlery health [--dir <path>] [--json]
knowlery query  "<question>" [--dir <path>] [--k <n>] [--json]
knowlery stale  [--dir <path>] [--json]
knowlery bundle install <zip-or-folder> [--dir <path>] [--force] [--skip-conformance]
knowlery bundle list      [--dir <path>] [--json]
knowlery bundle uninstall <bundle-id> [--dir <path>]
```

- `--dir` defaults to the current directory everywhere.
- **`init`** prompts for platform and name when flags are omitted (interactive
  terminals only; scripts must pass flags). Refuses an initialized workspace unless
  `--force` (which re-runs the full setup; notes and custom skills are preserved).
- **`sync`** applies exactly the updates the plugin applies on upgrade and lists the
  changed files; "No changes" means the workspace is current. Exit 1 with an upgrade
  hint if a newer Knowlery synced the workspace last.
- **`health`** exits 0/1 — usable directly as a CI gate.
- **`query`/`stale`** are read-only; abstention (`No confident matches ...`) exits 0
  because absence of knowledge is a result, not an error.
- **`bundle install`** prints the manifest summary and conformance outcome; blocked
  installs name the flag that overrides.

### Workflows

**Start in a terminal, graduate to Obsidian**

```bash
mkdir ~/notes/kb && knowlery init --dir ~/notes/kb
# write notes, let your agent /cook them...
knowlery query --dir ~/notes/kb "what do I know about X?"
```

Later, open `~/notes/kb` in Obsidian: the plugin recognizes it as fully initialized —
dashboard, Knowledge health, and in-app retrieval work immediately.

**Existing Obsidian vault + CLI on a second machine / server**

```bash
knowlery sync --dir /path/to/vault     # usually "No changes"
knowlery query --dir /path/to/vault "..."
```

**Cloud agent / CI**

```bash
knowlery health --dir "$VAULT" || exit 1     # gate
knowlery stale --dir "$VAULT" --json          # feed /cook's scope
knowlery query --dir "$VAULT" "$QUESTION" --json
```

Agents working *inside* a vault without a global install use the always-present
embedded script instead: `node .knowlery/bin/query.mjs "..."`.

**Receive a shared knowledge bundle**

```bash
knowlery bundle install ~/Downloads/team-onboarding.zip --dir ~/notes/kb
knowlery query --dir ~/notes/kb "<topic from the bundle>"   # bundle knowledge retrieves immediately
```

### FAQ / troubleshooting

| Question | Answer |
|---|---|
| Does the CLI need Obsidian installed? | No. Nothing in the CLI touches Obsidian. |
| Can plugin and CLI touch the same vault? | Yes — all writes are content-compared and the sync implementation is shared. The downgrade guard protects against *version skew*: update the older shell if sync refuses. |
| Why does `bundle export` not exist? | The approve/flag review gate is interactive by design; a headless export form is planned (0.8). Export from the plugin; install anywhere. |
| `sync` says a newer Knowlery synced this workspace | Another machine/shell runs a newer version. `npm i -g knowlery@latest` (or update the plugin) and retry. |
| What exactly does npm install? | One file: `knowlery-cli.mjs` (plus README/LICENSE). The Obsidian plugin is never in the npm package. |
| Where does CLI state live? | Nowhere outside the vault. All state is files in the workspace — same as the plugin. |
